SODA Query Examples

Here's a collection of SODA-query examples. These queries assume that there's a Pilot-class with a name and age, a Car class with a pilot and name and a BlogPost class with list of tags and a of meta-data.

There are also a few cases where SODA has special behavior for queries.

Type Constraint

This is most basic and most used constraint for SODA-queries. SODA acts like a filter on all stored objects. But usually you're only interested for instances of a certain type. Therefore you need to constrain the type of the result.

Field Constraint

You can add constrains on fields. This is done by descending into the field and constrain the value of the field. By default the constrain is an equality comparison.

Comparisons

You can do comparison on the field-values. For example to check if something is greater or smaller than something else.

Combination of Constraints (AND, OR)

You con combine different constraints with an 'AND' or 'OR' condition. By default all constrains are combined with the 'AND' condition.

Not-Constrain

You can invert a constrain by calling the not-constrain.

String Comparison

There are special compare operations for strings. By default strings are compared by equality and the comparison is case sensitive.

There's the contains-comparison which checks if a field contains a substring. The like-comparison is the case-insensitive version of the contains-comparison.

Also a start-with- and a ends-with-caparison is available for strings. For this you can specify if the comparison is case sensitive or not.

Compare Field With Existing Object

When you have a reference type field, you can compare this field with a certain object. It will compare the field and the object by object identity.

Note that this comparison only works with stored objects. If you do this with a not yet stored object, it will use query by example. To force the comparison by the object identity, you can add a .Identiy() call.

Descend Deeper Into Objects

You can descend deeper into the objects by following fields. This allows you to setup complex constraints on nested objects. Note that the deeper you descend into the objects, the more expensive the query is to execute.
