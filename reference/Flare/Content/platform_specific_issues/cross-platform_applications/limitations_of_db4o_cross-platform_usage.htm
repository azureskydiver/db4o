Limitations Of db4o Cross-Platform Usage

The db4o cross-platform functionality is currently quite limited. It provides the basic features that enable you to use Java database on .NET and vice versa. However, it is recommended to familiarize yourself with the current limitations. Some objects are treated differently in Java and .NET and can not be translated cleanly.

Limitations on Types

  • Enumerations. In Java enumerations are similar to classes whereas in .NET enumeration is just a primitive type. This means that you cannot represent a Java-enum as .NET-enum and vice versa. As for now you should avoid enums.
  • You cannot use collections. db4o uses special collections-type handlers to make collections more efficient. Those handlers are not ready for cross platform usage. You need to use arrays.
  • You cannot give a field-name an alias. You need to use the same field-names.
  • Generics are completely different represented on the two platforms. Avoid generic classes.
  • Some primitives are different. The byte is different on the two platforms. The Java byte is signed where as the .NET byte isn't. This leads almost certainly to confusion. Therefore you should avoid using bytes in a cross-platform scenario.
  • .NET uint, ulong and decimal have no equivalent on the Java-platform.

In general when your using db4o in a cross-platform environment, you should try to keep your persistent class definitions simple and unambiguously interpretable on both platforms. Avoid constructs that exist only on Java or only on .NET.

Additional Limitations

In a cross platform client server scenario, the server needs to know the classes. You cannot run a server without the classes deployed and access it with a client from another platform. You need a server which has all the persisted classes and then use aliases to those classes.
