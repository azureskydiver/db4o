Implementation

This topic explains how transparent activation/persistence is implemented and how you can implement the required interfaces by hand. In practice you should use the provider enhancer's instead of implementing the interfaces yourself. See "Transparent Activation/Persistence Example"

Adding the Configuration

We need to explicitly configure transparent activation/persistence.

Transparent Activation

This adds transparent activation, which automatically activates objects.

Transparent Persistence

This adds transparent persistence, which automatically activates objects and manages changes on objects. Includes implicitly the transparent activation support.

Optionally you can specify how rollbacks are handled.

Implementing the Activatable Interface

In order to support Transparent Activation, the objects which are stored in the database need to implement the -interface.

Implementing the -interface manually for every class is repetitive and error prone. That's why this process can be automated. See "TA Enhanced Example"

Now transparent activation/persistence is enabled. Now you can navigate the object-graph as deep as you want. The transparent activation will load the objects from the database as you need them. When you've enabled transparent persistence updates are also done transparently.

Behavior in Mixed Mode

In some environments there are both, objects which implement the -interface on other which don't. What's the behavior in this scenario? Then the behavior is this:

  • Objects which implement the -interface are activated when they are used the first time.
  • Objects which do not implement the -interface are always fully loaded.

That behavior ensures, that run into not activated objects in the transparent activation / persistence mode.

However objects which do not implement the -interface are not updated with transparent persistence. You have to do that yourself or enhance those classes aswell.
